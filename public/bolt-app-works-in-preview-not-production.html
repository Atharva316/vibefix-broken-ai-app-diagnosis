Bolt App Works in Preview But Not Production — First Checks

If your Bolt app works in preview but breaks after deploy, the problem is often not the UI. It may be environment variables, API URLs, auth redirects, build settings, or production-only behavior.

Check these first

  • Are environment variables set in production?
  • Are API URLs different between preview and production?
  • Are auth redirect URLs updated?
  • Are database permissions different?
  • Does the build log show warnings or failed assets?
  • Are browser console errors different in production?

Do not do this

  • Do not rewrite the whole app.
  • Do not reset your database blindly.
  • Do not change auth settings without knowing the redirect issue.
  • Do not keep prompting “fix production” without logs.

The 5 most common causes

  1. Environment variables not set in Bolt production settings. This is the first thing to check when a Bolt app works in preview but fails after deployment. Preview can use one configuration path while the production build depends on variables saved separately in the deployment environment. Missing API URLs, Supabase keys, auth callback URLs, payment webhook values, or service endpoints can create blank screens, failed requests, or login loops. You can identify this by comparing the working preview request against the failed production request in the browser network tab. This is different from a general app bug because the code may be fine. The live environment simply does not have the values the code expects. Before asking Bolt to rewrite components, list every required environment variable, confirm the exact names, and verify they are present in production.
  2. Hardcoded localhost:3000 URLs in API calls. AI-generated apps often begin with local URLs during preview and forget to replace them before production. The app can look perfect in Bolt preview, then fail on the live domain because the browser is still trying to call `localhost:3000`, a local API server, or a preview-only endpoint. Identify this by opening the production network tab and looking at failed request URLs. If any request points to localhost, a private port, or a development-only host, the issue is not the UI. It is an environment boundary mistake. This differs from a general backend bug because production users cannot reach your local machine. The safe fix is to replace hardcoded URLs with environment-based base URLs and confirm each environment points to the correct API.
  3. Build optimization removing code that worked in dev. Production builds minify, tree-shake, bundle, and optimize code differently than preview. Code that accidentally depends on dynamic imports, browser-only globals, unused-looking exports, or unsafe conditional class names can survive development but fail after optimization. The symptoms include missing components, styles that disappear, code that runs in preview but not production, or errors that only appear in the deployed build. This is different from a general bug because the source behavior and production bundle behavior diverge. Identify it by checking the production build logs, looking for warnings, and testing whether the failure appears only after deployment. Do not ask Bolt to redesign the page first. Ask it to inspect build-only differences and find the smallest code path affected by optimization.
  4. CORS policy not updated for the production domain. A Bolt preview URL may be allowed by your backend or third-party service while the final production domain is blocked. The browser then refuses API responses even when the server is reachable. Look for console errors mentioning CORS, blocked origins, preflight requests, or missing `Access-Control-Allow-Origin` headers. This is not a general API bug because the endpoint can work in preview, Postman, or server-side tests while failing in the production browser. The difference is the requesting origin. Check your backend, Supabase function, Firebase function, custom API, or third-party dashboard for allowed domains. Add only the correct production domain. Avoid disabling CORS broadly or accepting every origin unless you fully understand the security tradeoff.
  5. Database connection string pointing to the dev database instead of production. Sometimes production deploys successfully but reads from the wrong database. The app may show old data, no data, missing users, failed migrations, or tables that do not match the latest schema. Identify it by checking which database host and project the production environment variables reference, then compare a known production record against what the app loads. This is different from a general database bug because the query may be correct and the live app may still connect successfully. It is just connected to the wrong place. Do not reset tables or rerun migrations blindly. First confirm the production connection string, project reference, schema version, and RLS policies. Only then decide whether to update configuration, run migrations, or roll back.
